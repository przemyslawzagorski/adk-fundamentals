FEATURE REQUEST — Notification Hub Service
==========================================
Project: CLM-6 / Polaris Platform
Requested by: Product Owner — Marek Wiśniewski
Date: 2026-03-10
Priority: High

## Background

Currently each microservice in the Polaris platform sends notifications independently:
- CLM Core sends email confirmations directly via SMTP
- Billing module sends SMS via a hardcoded Twilio integration
- Reporting module sends push notifications via Firebase (deprecated SDK)

This leads to:
- Duplicated notification logic across 6+ services
- No unified audit trail of sent notifications
- Impossible to throttle or deduplicate notifications per user
- No retry mechanism on failures
- Marketing cannot manage notification templates without a developer

## Goal

Build a centralized Notification Hub microservice that:
1. Accepts notification requests from internal services via REST API or message queue
2. Routes to the appropriate channel: EMAIL, SMS, PUSH, IN-APP
3. Supports templating with dynamic variables (e.g. {{user_name}}, {{invoice_id}})
4. Provides a delivery status tracking endpoint
5. Handles retries on transient failures

## Known Integrations

- Email provider: SendGrid (we have an existing account, API key in Vault)
- SMS: Undecided — currently Twilio but contract expires Q3 2026. AWS SNS is being evaluated.
- Push notifications: Firebase Cloud Messaging — migration from deprecated SDK needed
- Internal services communicate via: REST + Kafka (some services only REST, some only Kafka)
- Template storage: TBD — could be DB, could be a CMS

## Consumers (Internal Services)

- CLM Core (Spring Boot, uses REST)
- Billing Module (Spring Boot, uses Kafka topics)
- Customer Portal (React frontend, uses WebSocket for in-app)
- Reporting Module (Python, uses REST)
- Mobile App (iOS + Android) — wants push notifications

## Team

- Backend: 2 Java developers (Spring Boot)
- 1 DevOps (Kubernetes on Azure AKS)
- No dedicated DBA — shared DB team

## Constraints

- Must be deployable on existing AKS cluster
- Existing services should require minimal changes to integrate
- GDPR compliance required — no PII in logs
- Must be ready for integration testing by Q2 2026

## Open Items (unresolved at feature request stage)

- SMS provider decision pending (Twilio vs AWS SNS)
- Template storage mechanism not decided
- No agreement yet on whether async (Kafka) or sync (REST) should be the primary interface
- No NFR targets defined yet (latency, throughput, availability)
- Authentication between internal services: mTLS, JWT, or API key — not decided
- Database: team prefers PostgreSQL but no decision on schema yet
